Arquivo ConfirmacaoPedido.htm

Caso este arquivo exista na pasta HTM da loja, ele será automaticamente incluído na etapa final, onde é exibido o pedido concluído.

Se a loja utiliza a forma de pagamento Bcash, o script de integração, que normalmente fica no arquivo ConfirmacaoPedido.htm, deve ser alterado. Clique aqui para ver os detalhes.

Também deverão ser feitos testes e adaptações (se necessárias) nas seguintes ocasiões:

- Caso a loja utilize códigos JavaScript personalizados

- Caso a loja tenha código que insere banner do e-bit ou algum outro design personalizado. Clique aqui para ver o código sugerido para exibir este banner.


Arquivos HTML para forma de pagamento adicional

Para fornecer maiores explicações ao cliente sobre a forma de pagamento adicional, é possível enviar arquivo HTML contendo detalhes sobre cada forma adicional ou programação JavaScript. Este arquivo será incluído na etapa onde é exibido o pedido concluído, junto à descrição da forma de pagamento escolhida pelo cliente.

Caso a loja utilize alguma programação especial JavaScript nestes arquivos, deverão ser feitos testes e adaptações se necessárias.


Implementação com Teste A/B

O Teste A/B é o método mais utilizado para realizar testes de usabilidade em lojas virtuais. Ele permite, por exemplo, comparar os resultados em vendas para duas versões de design do site, a versão A e a versão B.

Para direcionar apenas uma parte dos visitantes (ex: 10%) para o novo checkout e assim comparar as vendas obtidas com cada tipo de checkout, ative o Xtreme Checkout e o recurso de Teste A/B. Crie um arquivo com o nome AbtCheckout.htm, que deverá ser enviado para a pasta HTM. Este arquivo conterá um redirecionamento, conforme o exemplo abaixo:

<script>
document.location.href="/recalcula.asp?idloja=<idloja>&comprar=1";
</script>
Desta forma os visitantes do design A irão efetuar os pedidos através no Xtreme Checkout e os visitantes do design B irão efetuar pedidos através do tradicional checkout passo-a-passo. Após um período com esta configuração (ex: 30 dias), utilize os relatórios de vendas para comparar os resultados obtidos com cada tipo de checkout. Se os 10% dos visitantes que foram direcionados para o Xtreme Checkout fizerem mais do que 10% do total de vendas da loja, então estará comprovado que vale a pena adotar o novo checkout em 100% das visitas.


Outras observações

- Ao ativar a opção de Xtreme Checkout, o recurso de Layout diferenciado nas páginas de pedido ficará ativo por padrão na loja. Este recurso remove elementos desnecessários do design das páginas de fechamento de pedidos. Desta forma é necessário o envio de arquivos específicos (Ped*.htm) caso estes ainda não tenham sido enviados. Veja mais detalhes aqui.

- O método JavaScript document.write não pode ser utilizado nos arquivos HTM utilizados no checkout, pois é conflitante com AJAX.

- Caso a loja tenha códigos JavaScript personalizados nos arquivos HTMLs da loja (Ex: Rodape.htm, PedRodape.htm, etc) estes devem ser verificados e adaptados caso necessário.

- Textos personalizados de frete e de pagamento devem ser verificados, pois podem não caber no design ou podem conter JavaScript não suportado.

- Textos personalizados dos campos Texto ao início do cadastro e Texto ao final do cadastro da página Textos & imagens do site administrativo devem ser verificados, pois podem não caber no design ou podem conter JavaScript não suportado.

- Campos adicionais de clientes e de pedidos com scripts de validação devem ser testados e adaptados se necessário.

- Caso a loja tenha configurado o funil do Google Analytics, a loja deverá alterar as etapas deste funil ou criar uma nova meta de funil com as etapas do Xtreme Checkout. Veja exemplo destas etapas.


Atenção: Como este recurso altera o processo de finalização de pedidos, faça testes com todas as formas de pagamento disponibilizadas pela loja, principalmente as formas personalizadas.
